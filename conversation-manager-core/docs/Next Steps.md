# Conversation Manager Next Steps

This document records the current maturity of `conversation-manager` and the next backend boundaries needed before it becomes the durable conversation store for `agent-runner`.

## Current State

`conversation-manager` is now a usable conversation-management backend skeleton.

The following areas are already in shape:

- Conversation lifecycle: create, read, list, rename, soft delete.
- Conversation groups: create, update, reorder, delete, add conversations, remove conversations.
- Root-list pinning: `conversation.pinned` is the source of truth, and grouped conversations cannot be pinned.
- Sharing and forking: shared conversations are snapshotted by `endMessageId`, and fork copies messages with a database-side `INSERT INTO ... SELECT ...`.
- Message storage model: `conversation_message` supports text, multimodal `contentParts`, tool calls, tool call IDs, agent IDs, and finish reasons.
- Auth boundary: downstream user identity comes from `auth-center-session` and propagated trusted headers.
- Persistence conventions: table-scoped MyBatis mappers, PostgreSQL triggers for `modification_time`, and service-local global exception handling.

This is enough for user-facing conversation organization and history reads, but not yet enough for complete agent runtime persistence.

## Main Missing Boundary

The key missing piece is the write protocol between `agent-runner` and `conversation-manager`.

`ConversationMessageMapper` already has low-level message insert methods, but the service, controller, and Dubbo API do not yet expose a clear way to persist an agent run. The next step is to define message-writing interfaces around conversation turns, not only raw table inserts.

## Proposed Write Interfaces

### `appendMessages`

Append one or more messages to an existing conversation.

Expected use cases:

- Save a user message submitted from the UI.
- Save assistant output after model completion.
- Save tool call and tool result messages.
- Save partial batches generated by `agent-runner`.

Important behavior:

- Validate conversation ownership through the current user context or trusted service context.
- Preserve message order in the same order as the request.
- Use the existing content format from [content-format.md](content-format.md).
- Update the parent conversation's modification time after successful append.

### `saveConversationTurn`

Persist a complete user-to-agent turn in one service call.

Expected use cases:

- UI sends a user message, `agent-runner` produces assistant messages and tool messages, and the whole turn is saved as one logical unit.
- Retry or failure handling can keep one turn boundary visible.

Important behavior:

- Accept the user message plus generated assistant/tool messages.
- Run in one transaction.
- Return the saved message IDs and updated conversation abstract.
- Leave room for future status fields such as running, failed, canceled, or completed.

### `createConversationIfAbsent`

Create a conversation only when the caller does not already have one.

Expected use cases:

- First user message starts a new conversation.
- Agent-runner can avoid a separate preflight call.

Important behavior:

- If `conversationId` is absent, create a new conversation and return it.
- If `conversationId` is present, validate ownership and reuse it.
- Keep the default title flow compatible with automatic title generation later.

### `touchConversation`

Update the parent conversation metadata after message writes.

Expected use cases:

- Move active conversations to the top of the root list.
- Record the user or service that last modified the conversation.

Important behavior:

- Avoid repeating manual `modification_time = NOW()` in SQL; rely on the PostgreSQL trigger.
- Keep pinned conversations pinned but still update their modification time.
- Do not touch deleted conversations.

### `updateTitleByFirstUserMessage`

Set a better initial title when the first user message arrives.

Expected use cases:

- Replace the default `New Conversation` title with a short title derived from the first message.
- Later, this can be replaced by an LLM-generated title.

Important behavior:

- Only update conversations still using the default title.
- Trim to the existing title length limit.
- Keep manual title edits from being overwritten.

## RPC Surface

The Dubbo API currently exposes only a small read/update subset:

- `updateTitle`
- `deleteMessages`
- `getConversationMessageHistory`

Before `agent-runner` depends on `conversation-manager`, add the write-oriented methods to `conversation-manager-api` so agent runtime code does not call HTTP controllers or DAOs directly.

Recommended additions:

- `createConversationIfAbsent`
- `appendMessages`
- `saveConversationTurn`
- `getConversationMessageHistory`
- `deleteMessages`

## HTTP Surface

HTTP endpoints should remain user-facing and gateway-routed.

Recommended additions:

- `POST /conversation/messages`
- `POST /conversation/turn`

These should delegate to the same service methods used by Dubbo, so HTTP and RPC behavior stays aligned.

## Validation Rules

The next implementation should keep these rules:

- All user writes must verify conversation ownership.
- A forked conversation belongs to the user who forks it.
- Deleted conversations cannot receive new messages.
- Message deletion must verify that the parent conversation belongs to the current user.
- Group membership affects root-list display and pin eligibility, but it should not change message history semantics.

## Persistence Rules

Use the existing PostgreSQL and MyBatis style:

- Keep mapper methods table-scoped.
- Prefer set-based SQL for batch writes.
- Use transactions for multi-message turn saves.
- Use database triggers for modification timestamps.
- Prefer `RETURNING` when the response needs database-generated fields.
- Store `contentParts` and `toolCalls` as JSON strings at the entity layer, while API DTOs expose structured objects.

## Acceptance Criteria

The next step is complete when:

- `agent-runner` can create or reuse a conversation through RPC.
- `agent-runner` can persist a complete conversation turn through RPC.
- The UI can append user messages through the gateway.
- Message history reads return the same structured content that was written.
- Parent conversation modification time changes after message append.
- Root-list ordering reflects pinned state first, then latest active conversations.
- Local verification starts `conversation-manager-core`, exercises the new endpoints/RPC-equivalent service flow, and stops all started processes.
